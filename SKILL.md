# SeestarAI 小红书推送

全自动将当日最重要的 AI 新闻转化为深度报道卡片 + 配套文案。

## 核心理念

不是堆新闻标题。每条新闻是一个独立的故事，结构由故事决定而非模板决定。不为了保险做一样的选题，不为了省事套一样的结构。

每一期的检验标准：这期跟上一期比，选题的角度、叙事的方式、版面的节奏——至少两个不一样。

---

## 流程总览

```
Phase 1: 新闻发现 → Phase 2: 选题与审查 → Phase 3: 叙事设计 → Phase 4: HTML 生成 → Phase 5: 文案输出
```

全程不中断询问用户。只在最终输出后接受反馈。

## Phase 1: 新闻发现

### 信源策略

同时覆盖三类信源，每类至少拉取 2 个：

| 类别 | 信源 | 方法 |
|------|------|------|
| 一手报道 | TechCrunch, The Verge, Ars Technica | WebFetch 抓 AI 频道首页 |
| 财经视角 | FT, Bloomberg, MoneyControl, Yahoo Finance | WebFetch 搜索页或直接 URL |
| 行业数据 | 公司官方博客, Ramp, StockAnalysis, OpenRouter | WebFetch 抓取 |

### 搜索策略

以当天日期为中心，并行 WebFetch。覆盖：AI 频道最新 10 条 + 至少 2 个财经信源 + 至少 1 个行业数据信源。

## Phase 2: 选题与审查

### 2.1 广撒网

从所有信源中列出所有可做的选题（不设上限），逐一标记：这条新闻的「核心张力」是什么？

「核心张力」不是"发生了什么"，而是"什么跟什么在对抗/拉扯/反转"：
- 预期 vs 现实（谁以为 X，结果 Y）
- 速度 vs 质量（增长太快/太慢带来的问题）
- 巨头 vs 新贵（格局在动）
- 技术 vs 商业（能用了 ≠ 能卖）
- 单点 vs 趋势（一次事件还是信号）

如果一条新闻找不到核心张力，降级为备选。

### 2.2 发散：三个角度

从候选选题中挑最有张力的 3 条，每条用一句话写出 3 个不同角度：

| | 角度 A（常识视角） | 角度 B（逆向视角） | 角度 C（放大视角） |
|---|---|---|---|
| 选题1 | 大多数人会怎么看 | 反过来看有什么忽略的 | 放大了看暗示什么趋势 |
| 选题2 | ... | ... | ... |
| 选题3 | ... | ... | ... |

选择标准：三个角度里，哪个写出后自己最想继续往下读。如果三个都平平，说明这个选题不值得做——换下一个。

### 2.3 反模板检查（必做）

在最终选定选题前，问自己：

1. 这个选题跟你上次做的像不像？如果像，角度不同吗？
2. 这个选题是不是"又一家公司融资了"或"又一家公司发布了新模型"——如果是，它真正不同的点在哪？
3. 如果不用任何量化数字，这篇文章还能成立吗？（如果不能，选题太薄）

### 2.4 数据审查（保留）

对选定选题中的每个数据声明执行三级验证：

- **Level 1 — 信源交叉**：至少 3 个独立信源，低于 2 个标注"置信度：中/低"
- **Level 2 — 原始报告追溯**：无法追溯标注"二手引用"
- **Level 3 — 上下文校准**：对比行业可比数据、获取最新市值、确认时点

审查输出：
```
| 数据点 | 值 | 信源1 | 信源2 | 信源3 | 置信度 |
```

## Phase 3: 叙事设计

**核心原则：不要先想"分几页"——先想"这个故事的结构是什么"。**

### 3.1 故事类型诊断

先判断这条新闻的本质叙事类型。不是贴标签——是帮你找到自然的讲述节奏。

| 故事类型 | 特征 | 自然流向 | 大概页数 |
|---------|------|---------|:--:|
| **转折点** | 一个数字/事件改变了行业格局 | 冲击 → 事件 → 对比（之前 vs 之后）→ 连锁反应 → 展望 | 8-10 |
| **加速** | 一个趋势在变快/变大 | 速度感 → 驱动力 → 阻力 → 谁受益谁受损 → 天花板在哪 | 7-9 |
| **碰撞** | 两股力量在对抗 | 冲突亮相 → 各自逻辑 → 数据对决 → 旁观者视角 → 未决 | 9-11 |
| **反转** | 预期被打破 | 常识/预期 → 意外 → 追问为什么 → 是孤例还是趋势 | 7-9 |
| **解锁** | 一个技术/规则变化打开了新可能 | 之前为什么锁着 → 哪里开了 → 谁先冲进去 → 接下来开什么 | 7-9 |
| **信号** | 一个不起眼的变化暗示大转折 | 细节 → 为什么被忽略 → 放大看 → 如果是对的意味着什么 | 8-10 |

以上是起点，不是框。一个选题可能同时是"加速+碰撞"，那就融合两种流向。

### 3.2 页面设计

根据故事类型决定页数和每页的角色。不预设页码配比——先列"这篇报道必须回答的问题"，每个问题一页。问题质量决定报道质量。

**页面节奏原则：**
- 相邻页不同类型的视觉组件（数字密集页 → 文字叙事页 → 图表对比页 → 引语页，交替）
- 长叙事后必跟数据或视觉页喘气
- 读者每翻一页要感到"有进展"，不是"又多了一页"
- 总页数 7-12 之间

**每页设计时问自己：**
- 这页如果删掉，读者会缺失什么？如果没缺失，删掉
- 这页跟上一页的区别是什么？如果没区别，合并
- 这页用什么组件讲最合适？（从 design-spec.md 选，但不为了用组件而用）

### 3.3 封面定调

封面不是标题总结。封面是钩子——读者 3 秒决定滑走还是继续。

封面三要素：
1. 一个核心数字或一个核心对比（冲击力）
2. 一句话说出这件事的新意（不是"发生了"，是"第一次/反超/逆转/打破了"）
3. 一个角度暗示（让读者觉得后面有料）

### 3.4 反模板检查（出结构后必做）

- 这期的页码顺序跟上一期一样吗？如果一样，调
- 这期用了上期没用过的组件吗？至少一个
- 这期的叙事类型跟上期不同吗？如果相同，换角度

## Phase 4: HTML 生成

### 4.1 设计规范

遵循 `references/design-spec.md`。核心约束：

- 卡片：540×720px（3:4），圆角 24px
- 字体：PingFang SC / Microsoft YaHei
- 品牌：左上角 "SeestarAI分享" + 右侧导航标签
- 配色：深色 `#1c1c1e` / 浅色 `#faf8f5`，强调色 `#c44a2a`（浅）/ `#d4744a`（深）
- 排版参数见 design-spec.md

### 4.2 内容约束

- 不溢出 540×720px
- 每页 2-4 个信息块
- 所有引语来自已验证信源原文
- 每页品牌 "SeestarAI分享" 可见
- 公司全名只在封面出现

### 4.3 生成方法

以 `assets/template.html` 的 CSS 框架为基础。CSS 不变，body 中 `.card` 按叙事设计重新组织——不套用模板的页码顺序，每页组件根据内容选择。

### 4.4 自检

1. 每页 540×720px 内？
2. 所有数字有来源？
3. 所有引语来自验证信源？
4. 品牌每页可见？
5. 公司名只在封面？
6. 这期跟上期的结构/节奏/视觉有明显差异？

## Phase 5: 文案输出

### 5.1 文案结构（灵活版）

```markdown
[标题：事件核心 + 一个矛盾/反差 + 🔥]

[导语：2-3 句，让不知道这事的人也感到"有意思"]

---

### [小标题：根据叙事自然分段，3-5 个小节]

[每节 2-3 句 + 关键数字，数字不在多在于准]

---

### 我的判断

[1-2 句，可以是疑问、可以是预判、可以不给出答案]

---

📸 滑动查看完整深度拆解
[3-5 个 hashtag]
```

### 5.2 文案风格

- 跟做投资的朋友聊天，不教书
- 🔥 只在标题用，别处不用
- 可以不下结论，但必须有立场
- Hashtag ≤5 个

## 附加规则

### 不能做的

- 不编造数据或引语
- 不让内容溢出 540×720px
- 不在非封面页重复公司全名
- 不跳过数据审查
- 不连续两期用相同叙事类型
- 不因为"上期这样做了"就默认为标准

### 必须做的

- 每期选至少一个上期没用过的布局组件
- 数据置信度用措辞区分（"已确认"/"据报道"/"业内估计"）
- 第 10 页（或最后页）列出全部数据来源
- 封面保留核心数字
